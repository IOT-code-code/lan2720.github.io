【Tensorflow6位验证码实战-3】如何构建model.py和train.py

发表于 2016-11-09   |     |   阅读次数

前言

在第二讲中，我们用TFRecords生成了训练所需的batch数据，这一讲的任务就是从零开始，完成模型部分model.py和训练部分train.py的代码。

哪些放在model.py中

首先，我们需要决定哪些代码要放在model.py中，哪些放在train.py中。
这个问题其实会困扰每一个刚入门tensorflow的同学。为什么这么说，因为不管是官方给出的样例代码，还是github上各式各样的项目代码，大家构建整个项目的方式都不太相同。以下是我总结的三种：

  1. 将模型代码单独提出到一个.py文件中，写成一个类，比如TextCNN。参考这里。整个类中有深度学习的各种结构，有output，loss，accuracy。和这里，train/valid/test分别创建三个实例来用。
  2. 模型和训练同在一个py文件。参考这里和这里。
  3. 还有一种更高级的结构，参考Danijar Hafner。

到底用哪种，每个人的喜好不一样。但是我觉得有几个基本点要把握：

  1. train/valid/test的时候都是有labels的，但是对于很多问题，训练好了模型，我希望给new input得到inference，这时候不知道其label，没法传入，因此labels不应该作为Graph中一个placeholder存在，因为placeholder在sess.run()的时候必须要以feed_dict传入。
  2. 基于第二讲我们从tfrecords中解析，然后获得的batch_images和batch_labels，其类型是Tensor Const，而做inference的时候，我们的new image经过预处理得到一个numpy array，我们希望这个array传入到model中，predict出一个array，那么很显然需要sess.run()，new image作为placeholder，以feed_dict传入到Graph。考虑到这点不同，个人认为input输入就不应该直接硬性的定义在model代码中了。因为要定义成一个placeholder吧，tfrecords得到的train_batch_data不能feed in；要不定义成placeholder吧，inference数据又没法传入。
  3. 根据以上两点，inputs和labels都不能定义在model中了，因为model中放的只能是公共部分，也就是train/test和inference的时候都要有的结构，例如：卷积层，全连接层等等。既然如此，output是公共部分，应该纳入model；但是loss，accuracy在inference的时候不存在，因此不应该纳入model；再就是prediction（一般prediction是对output进行argmax()，即predict出的label），这个在train阶段不是必须的，因为train更关心loss（要不断optimize loss），loss的计算只需要output和labels即可。

结构图

根据这个想法，简单画了个图，可以说整个图的内容是对tensorflow Graph的一个抽象。

  1. 中间的model.py作为一个整体封装成一个部件。
  2. 周边的inputs，labels，loss等结构是在train.py中加到Graph上的，train.py中维护一个全局tf.Graph()。
  3. 每一部分都可以作为一个零部件，写代码时注意良好的可拔插性。
  4. model.py对外的接口只有inputs和outputs。
  5. labels对于不同的功能，相应传入

好了，这样说来对整个structure的认识应该清晰很多。

完成model.py（项目中命名为captcha_cnn.py）

想法仍然是在model.py中完成一个CaptchaCNN类。之所以要实现一个类而不是一个函数，是因为模型结构有一些参数需要设置和定义，保存着全局都要用，函数无法实现这样的要求。

__init__()函数

__init__()应该传入构建CNN的一些必要参数，例如：

  • filter_sizes：是一个list，每层都有一个filter_size，如第一层filter_size=3(3*3的小正方形)，第二层=2(2*2)，诸如此类。。如[3, 2, …]
  • num_of_filters：也是一个list，定义每一个卷积层有多少个filters，如[64, 64, …]
  • filter_strides：同上
  • pool_sizes：仍然是一个list，每一个层进行max pool时的窗口大小，同样是正方形只需给出边长
  • pool_strides：同上
  • pool_types：pool type指的是每一池化层的类型，比如有maxpool和avgpool两种
  • input_channels：是1还是3，在第一个卷积层需要用到这个值
  • hidden_sizes：也是一个list，比如在多个conv-maxpool之后再连接2个全连接层，那么给出每一层的隐层单元数
  • num_of_labels：multi-label问题，比如一张图中既可能有动物，又可能有人，植物出现的时候，该参数就=3；验证码中有6位字符，该值=6；如果是exclusive classification，该值=1。
  • num_of_classes：每种label可能出现的情况。比如人，可能在画面中出现多少种人。验证码每一位字符可能出现a-zA-Z0-9共62种情况，那么该值=62

多个卷积层之后就是全连接层，常规结构，直接上代码：

可以看到我写了两部分，一个是带激活函数的，一个不带。为什么这么写，是考虑到输出层。因为输出层也是全连接的，但是它不应该被relu，softmax等激活，而是logits传出，这样在传递出去后，loss部分的代码才进行softmax和cross entropy的计算。
这么做也是基于tensorflow的api中有这么一个函数tf.nn.sparse_softmax_cross_entropy_with_logits(logits, labels, name=None)，我从文档中摘出一段warning，

WARNING: This op expects unscaled logits, since it performs a softmax on logits internally for efficiency. Do not call this op with the output of softmax, as it will produce incorrect results.

也就是说tensorflow内部在此处做了优化，因此只接收unscaled logits，不然就错了。所以，既然我要使用这个函数的话，outputs传出的时候就应该是logits的未被激活状态，因此要用到full_connect_layer而不是full_connect_acrivate。

以上两个函数就是基本模块了。

需要注意的是，由于是multi-label，网络需要同时学习多个label，因此有几个label（num_of_labels），就构建几个子输出层。这些子输出层的神经元个数就是num_of_classes，刚才我们传入到CaptchaCNN的这些参数也就是全用上了。子输出层最后concat起来，作为这个model的outputs。

train.py该写些什么

model.py中的代码弄清楚之后就会发现，真正能用于训练的graph还差一些结构，比如loss没有定义，accuracy没有写，optimizer也没有定义。很好，这些东西就需要在train.py中一点点的加到tf.Graph()上，和刚才写的model拼接起来，最后就能完整的构建一个可以用于train/inference的tensorflow graph了。训练部分的完整代码在这里。

自认为结构很清晰的，捋一下结构：

  1. FLAGS定义必要的参数
  2. 如果有TFRecords文件存在则跳过这步，没有则从原始pickle数据生成.tfrecords文件
  3. 用tf.shuffle_batch生成train batch data和labels（这部分代码在captcha_input.py中）
  4. 定义CaptchaCNN模型
  5. 传给模型inputs，flow()得到outputs，与此同时将这部分结构加载到Graph上
  6. 给Graph加上零部件：loss，predictions，accuracy。
  7. 定义optimizer，learning rate自动decay
  8. 最后是模型保存Saver，训练过程记录Summary等一些小结构
  9. 训练部分使用Coordinator，自动根据num_epochs来做数据生成，且多线程

That’s all! 我们完成了用tensorflow实现cnn来识别6位验证码的全部工作。

结果

GTX 1080上训练1个小时达到98%以上的accuracy，现在的代码我还没有加任何regularization的部分，所以能达到这么高是完全拟合了15万训练集数据。之后我还有几项TODO要做：
